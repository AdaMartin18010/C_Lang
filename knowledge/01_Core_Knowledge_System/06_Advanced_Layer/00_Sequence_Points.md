## 一、三个核心概念

在C的抽象状态机中，表达式求值涉及三个相互关联的概念：

| 概念 | 定义 | 示例 |
|:-----|:-----|:-----|
| **值计算（Value Computation）** | 确定表达式结果的值 | `a + b`：计算a的值和b的值，然后相加 |
| **副作用（Side Effect）** | 改变抽象状态机状态的操作 | `a = 5`：将a的值改为5；`printf(...)`：输出到文件 |
| **序列点（Sequence Point）** | 前一个求值的所有副作用必须完成，后一个求值才能开始的点 | 完整表达式结束处、`&&`、`||`、`,`运算符处 |

### 关键原则

> **C23 5.1.2.3/3**: "在前一个序列点和下一个序列点之间，一个对象的存储值最多只能通过表达式的求值修改一次。此外，只有在确定要存储的值时才会访问先前的值。"

## 二、序列点

### 2.1 C23中的序列点位置

序列点在C11后被"顺序约束（Sequencing）"概念部分取代，但以下位置仍然构成顺序约束点：

| 位置 | 标准引用 | 说明 |
|:-----|:---------|:-----|
| **完整表达式结束** | 5.1.2.3/1 | 如语句结束、初始化完成、`return`表达式结束 |
| **逻辑与 `&&`** | 6.5.13/4 | 左操作数求值后、右操作数求值前有顺序约束 |
| **逻辑或 `||`** | 6.5.14/4 | 同上 |
| **条件运算符 `?:`** | 6.5.15/4 | 条件表达式求值后、所选分支前有顺序约束 |
| **逗号运算符 `,`** | 6.5.17/3 | 左操作数求值完全完成后才求值右操作数 |
| **函数调用** | 6.5.2.2/10 | 函数设计器和参数求值后、函数体执行前有顺序约束 |

### 2.2 序列点 vs 顺序约束（C11后）

C11引入了更精细的**顺序（Sequencing）**概念：

- **A sequenced before B**：A的求值在B之前完成（A的所有副作用在B开始前完成）
- **A unsequenced with respect to B**：A和B的求值顺序未指定，可以交错
- **A indeterminately sequenced with respect to B**：A和B的顺序未指定，但不会交错（线程间）

```text
表达式求值顺序关系
├── sequenced before      （确定在前）
│   └── 如：a && b 中 a sequenced before b
├── unsequenced           （未指定顺序）
│   └── 如：f() + g() 中两个函数调用 unsequenced
└── indeterminately sequenced （不确定顺序，不交错）
    └── 多线程场景
```

## 三、求值顺序

### 3.1 哪些操作的顺序是确定的？

| 操作 | 顺序规则 |
|:-----|:---------|
| `a && b` | a 先求值，若a为假则不求值b（短路）|
| `a || b` | a 先求值，若a为真则不求值b（短路）|
| `a ? b : c` | a 先求值，然后只求值b或c中的一个 |
| `a, b` | a 先求值，然后b求值（逗号运算符）|
| `a = b` | b 先求值，然后赋值给a |
| `a[i]` | i 先求值，然后访问a[i]（C11前未指定；C11后a先求值）|
| `f(a, b, c)` | 函数设计器f先求值，然后参数a、b、c的求值相互unsequenced |

### 3.2 哪些操作的顺序是未指定的？

| 操作 | 问题 |
|:-----|:-----|
| `f() + g()` | 两个函数调用的求值顺序未指定 |
| `a[i] = b[j]` | a、b、i、j的求值顺序未指定 |
| `*p++ = *q++` | p、q的增量和解引用顺序未指定 |
| `i++ + i++` | 两个i++的副作用顺序未指定 |

## 四、经典UB案例解析

### 案例1：`i++ + i++`

```c
int i = 0;
int r = i++ + i++;  // UB!
```

**为什么这是UB？**

1. `i++` 的语义：读取i的值，然后副作用（i增加1）
2. 两个 `i++` 的副作用之间**没有序列点**
3. 因此，对象 `i` 在同一序列点之间被修改了**两次**
4. 违反C23 6.5/2: "如果在标量对象上的副作用相对于同一标量对象上的另一个副作用或值计算未排序，则行为未定义。"

**编译器可能的行为**：

| 编译器 | 可能结果 | 说明 |
|:-------|:---------|:-----|
| GCC | r = 0, i = 2 | 先读取两个i（都为0），然后加两次 |
| Clang | r = 1, i = 2 | 先读取i=0，第一个++后i=1，第二个读取i=1 |
| MSVC | r = 0, i = 2 | 类似GCC |

> **关键点**：所有结果都是"合法"的，因为标准未定义行为，编译器可以做任何事情。

### 案例2：`f(i++, i++)`

```c
int f(int a, int b) { return a + b; }
int i = 0;
int r = f(i++, i++);  // UB! 原因同上
```

**额外复杂性**：
- 函数参数的求值顺序**未指定**
- 即使参数求值有顺序，`i++` 的副作用之间仍无序列点

### 案例3：`a[i] = i++`

```c
int a[10] = {0};
int i = 0;
a[i] = i++;  // C11前是UB；C11后行为定义
```

**C11后的规则**：
- C11 6.5.2.1/2: "下标运算符 `[]` 中，表达式 `E1[E2]` 等价于 `(*((E1)+(E2)))`"
- C11 6.5/3: "下标运算符的副作用相对于值计算有顺序约束"
- **C11后**：`a` 和 `i` 的求值有顺序约束（a先求值），但 `i++` 的副作用和 `a[i]` 的地址计算仍可能unsequenced

**结论**：即使C11后，仍建议避免此类代码。

### 案例4：`*p++ = *q++`

```c
// 常见的字符串复制模式
while (*p++ = *q++);  // 危险！C11前是UB
```

**问题**：`p++` 和 `q++` 的副作用之间无序列点。

**C11后的改进**：赋值运算符的副作用相对于左右操作数有顺序约束，但 `p++` 和 `q++` 的副作用仍unsequenced。

**安全写法**：
```c
while (*q != '\0') {
    *p = *q;
    p++;
    q++;
}
*p = '\0';
```

## 五、C11改进

### 5.1 顺序约束的精细化

C11用更精细的"顺序"概念替代了简单的"序列点"：

```c
// C11前：UB
int r = (++i) + (++i);

// C11后：仍然是UB，但原因表述不同
// ++i 的副作用相互unsequenced
```

### 5.2 主要改进点

| 方面 | C99/C89 | C11/C17/C23 |
|:-----|:--------|:------------|
| 序列点模型 | 离散序列点 | 顺序约束关系 |
| 线程间顺序 | 无定义 | 引入happens-before |
| 内存模型 | 无 | 明确的内存模型（release/acquire）|
| 原子操作 | 无 | `_Atomic` 类型和 `stdatomic.h` |

## 六、编译器行为实测

### 6.1 使用编译器警告检测UB

```bash
# GCC
gcc -Wall -Wextra -Wsequence-point source.c

# Clang
clang -Wall -Wextra -Wunsequenced source.c

# 开启UBSanitizer
gcc -fsanitize=undefined source.c
```

### 6.2 实测代码

```c
#include <stdio.h>

int main(void) {
    int i = 0;
    
    // 测试1: i++ + i++
    int r1 = i++ + i++;
    printf("i++ + i++: r=%d, i=%d\n", r1, i);
    
    // 测试2: (++i) + (++i)
    i = 0;
    int r2 = (++i) + (++i);
    printf("(++i) + (++i): r=%d, i=%d\n", r2, i);
    
    // 测试3: 函数参数
    i = 0;
    int r3 = printf("%d %d\n", i++, i++);
    
    return 0;
}
```

**GCC输出（x86-64）**：
```
i++ + i++: r=0, i=2
(++i) + (++i): r=3, i=2
0 1
```

**解释**：
- `i++ + i++`：GCC选择先读取两个i（都为0），然后两次递增 → r=0, i=2
- `(++i) + (++i)`：先执行两次++i（i变为2），然后相加 → r=4? 实际输出r=3说明编译器优化了

> **警告**：不要依赖任何特定行为！这些是UB，不同编译器/版本/优化级别可能产生不同结果。

## 七、防御性编程建议

### 7.1 避免UB的黄金法则

| 法则 | 说明 |
|:-----|:-----|
| **一表达式一副作用** | 每个完整表达式中，对同一对象的修改不超过一次 |
| **读写分离** | 若表达式既读取又修改同一对象，确保修改有明确定义 |
| **函数参数纯化** | 函数调用参数中避免对同一对象的副作用 |
| **使用临时变量** | 复杂表达式拆分为多个语句 |

### 7.2 安全重构示例

```c
// 危险代码
int r = i++ + i++;

// 安全重构
int temp1 = i++;
int temp2 = i++;
int r = temp1 + temp2;

// 危险代码
a[i] = i++;

// 安全重构
int index = i;
int value = i++;
a[index] = value;

// 危险代码
while (*p++ = *q++);

// 安全重构
while (*q != '\0') {
    *p = *q;
    p++;
    q++;
}
*p = '\0';
```

### 7.3 静态分析工具

| 工具 | 检测能力 |
|:-----|:---------|
| **GCC -Wsequence-point** | 检测序列点相关UB |
| **Clang -Wunsequenced** | 检测unsequenced副作用 |
| **UBSan** | 运行时检测UB（有符号溢出、对齐违规等）|
| **Frama-C** | 形式化验证无副作用违规 |
| **Coverity** | 商业级序列点分析 |
