## 一致性模型形式化

### 1.1 线性一致性

**定义1.1 (线性一致性)**:
```
历史H是线性的，如果存在串行历史S使得：
1. S是H的排列
2. S保持H的实时顺序
3. S满足规范
```

**定理1.1 (线性一致性组合)**:
```
若对象O₁和O₂都是线性的，
则组合系统(O₁, O₂)也是线性的
```

### 1.2 顺序一致性

**定义1.2 (顺序一致性)**:
```
历史H是顺序一致的，如果存在串行历史S使得：
1. S是H的排列
2. S保持每个进程的Program Order
```

**关联**: 比线性一致性弱，但保持顺序性

## 共识算法

### 2.1 Paxos形式化

**定理2.1 (Paxos安全性)**:
```
不存在两个不同值被不同学习者接受
```

**证明要点**:
- 提议者必须获得多数接受者的承诺
- 接受者只能接受更高编号的值
- 多数集合相交保证一致性

### 2.2 Raft共识

**定理2.2 (Raft Leader完备性)**:
```
若日志条目在任期T被提交，
则所有更高任期的Leader都包含该条目
```

**状态机安全**: 所有节点按相同顺序应用相同命令

## 拜占庭容错

### 3.1 BFT界限

**定理3.1 (拜占庭容错下界)**:
```
容忍f个拜占庭故障需要至少3f+1个节点
```

**证明**: 
- 分区成3组: f个故障节点, f和f个正常节点
- 正常节点间无法区分故障与分区
- 需要多数派(2f+1)才能决定

### 3.2 PBFT算法

**定理3.2 (PBFT安全性)**:
```
若消息被提交，则所有诚实节点同意该消息
```

**三阶段协议**: 
- Pre-prepare: Leader提议
- Prepare: 2f+1个节点准备
- Commit: 2f+1个节点提交

## 分布式事务

### 4.1 两阶段提交

**定理4.1 (2PC原子性)**:
```
所有参与者都提交或都中止
```

**阻塞问题**: Coordinator故障导致阻塞

### 4.2 三阶段提交

**定理4.2 (3PC非阻塞)**:
```
在同步网络假设下，3PC是非阻塞的
```

**代价**: 额外一轮消息

## 复制与一致性

### 5.1 CAP定理

**定理5.1 (CAP不可能性)**:
```
分布式系统不能同时保证：
- 一致性(Consistency)
- 可用性(Availability)  
- 分区容忍性(Partition tolerance)
```

**选择**: CP系统或AP系统

### 5.2 最终一致性

**定理5.2 (收敛性)**:
```
在无新更新的情况下，
所有副本最终收敛到相同值
```
