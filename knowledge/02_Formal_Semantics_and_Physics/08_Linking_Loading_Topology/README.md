## 编译流程回顾

### 四阶段编译模型

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   源代码     │───▶│   预处理     │───▶│    编译      │───▶│    汇编      │
│   .c/.h     │    │   宏展开     │    │  生成汇编    │    │  生成目标文件 │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                                               │
                                                               ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────────────────────┐
│  可执行文件  │◀───│    链接      │◀───│        目标文件 .o/.obj      │
│  a.out/ELF  │    │  符号解析    │    │   (二进制 + 符号表 + 重定位)  │
└─────────────┘    └─────────────┘    └─────────────────────────────┘
```

### 各阶段详解

```bash
# 预处理阶段 - 处理宏、头文件包含、条件编译
gcc -E hello.c -o hello.i

# 编译阶段 - 将预处理后的代码转为汇编
gcc -S hello.i -o hello.s

# 汇编阶段 - 将汇编转为机器码
gcc -c hello.s -o hello.o

# 链接阶段 - 合并目标文件和库
gcc hello.o -o hello
```

### 预处理器的实际工作

```c
// 原始代码
#define MAX_SIZE 1024
#define SQUARE(x) ((x) * (x))

int buffer[MAX_SIZE];
int result = SQUARE(5 + 3);

// 预处理后
int buffer[1024];
int result = ((5 + 3) * (5 + 3));
```

## 目标文件格式

### 三种主流格式对比

| 特性 | ELF (Executable and Linkable Format) | COFF (Common Object File Format) | Mach-O (Mach Object) |
|------|--------------------------------------|----------------------------------|----------------------|
| 使用系统 | Linux, BSD, 嵌入式系统 | Windows (PE基于COFF) | macOS, iOS |
| 结构灵活性 | 高 | 中 | 高 |
| 节区数量 | 动态可变 | 固定限制 | 动态可变 |
| 调试信息支持 | DWARF | CodeView | DWARF |

### ELF 文件结构详解

```
┌─────────────────────────────────────────────────────────────────┐
│                         ELF Header (64字节)                      │
│  ├─ Magic: 0x7F 'E' 'L' 'F'                                      │
│  ├─ Class: 32/64位                                               │
│  ├─ Data: 小端/大端                                              │
│  ├─ Version: 版本号                                              │
│  ├─ OS/ABI: 目标操作系统                                          │
│  ├─ Type: ET_REL/ET_EXEC/ET_DYN                                  │
│  ├─ Machine: 目标架构 (x86, ARM, ...)                            │
│  └─ Entry point: 程序入口地址                                     │
├─────────────────────────────────────────────────────────────────┤
│                      Program Header Table                        │
│  描述运行时加载的段 (Segment) 信息                                │
├─────────────────────────────────────────────────────────────────┤
│                      Section Header Table                        │
│  描述链接时使用的节 (Section) 信息                                │
├─────────────────────────────────────────────────────────────────┤
│  .text    │  可执行代码                                          │
│  .data    │  已初始化全局/静态变量                                │
│  .rodata  │  只读数据 (字符串常量等)                              │
│  .bss     │  未初始化全局/静态变量 (不占文件空间)                  │
│  .symtab  │  符号表                                             │
│  .strtab  │  字符串表                                           │
│  .rel.*   │  重定位表                                           │
│  .debug   │  调试信息                                           │
└─────────────────────────────────────────────────────────────────┘
```

### 使用 readelf 和 objdump 分析

```bash
# 查看 ELF 头信息
readelf -h hello.o

# 查看节头表
readelf -S hello.o

# 查看符号表
readelf -s hello.o

# 反汇编代码
objdump -d hello.o

# 查看重定位条目
objdump -r hello.o
```

### 目标文件中的符号类型

```
┌─────────────────────────────────────────────────────────┐
│                     符号表结构                           │
├─────────────────────────────────────────────────────────┤
│  符号名      │  类型      │  绑定属性  │  所在节区  │  值  │
├─────────────┼───────────┼───────────┼───────────┼─────┤
│  main       │  FUNC     │  GLOBAL   │  .text    │ 0x0 │
│  printf     │  FUNC     │  GLOBAL   │  UNDEF    │ 0x0 │ ← 未定义符号

│  .rodata    │  SECTION  │  LOCAL    │  .rodata  │ 0x0 │
└─────────────────────────────────────────────────────────┘
```

## 符号解析与重定位

### 符号解析过程

```
           ┌─────────────────┐
           │   符号引用发现   │
           │  (编译阶段标记)  │
           └────────┬────────┘
                    ▼
    ┌───────────────────────────────┐
    │      链接器符号解析算法        │
    │  ┌─────────────────────────┐  │
    │  │ 1. 收集所有输入文件的    │  │
    │  │    符号定义和引用       │  │
    │  └─────────────────────────┘  │
    │  ┌─────────────────────────┐  │
    │  │ 2. 处理强符号/弱符号     │  │
    │  │    (属性冲突检测)       │  │
    │  └─────────────────────────┘  │
    │  ┌─────────────────────────┐  │
    │  │ 3. 解析库中的符号        │  │
    │  │    (从左到右扫描)       │  │
    │  └─────────────────────────┘  │
    │  ┌─────────────────────────┐  │
    │  │ 4. 报告未定义符号        │  │
    │  │    (链接错误)           │  │
    │  └─────────────────────────┘  │
    └───────────────────────────────┘
                    ▼
           ┌─────────────────┐
           │    重定位计算    │
           │  (修改代码/数据) │
           └─────────────────┘
```

### 重定位类型详解

```c
// 假设在 0x1000 处的代码需要引用 0x2000 处的变量

// R_X86_64_PC32 - 相对寻址重定位
// 计算: S + A - P
// S = 符号值 (0x2000), A = 加数, P = 重定位位置 (0x1005)
call printf      // 需要填充为 printf 地址 - 当前地址

// R_X86_64_32 - 绝对寻址重定位
// 计算: S + A
movl $global_var, %eax   // 需要填充为 global_var 的绝对地址

// R_X86_64_PLT32 - PLT 条目重定位 (用于动态链接)
call printf@PLT  // 跳转到过程链接表
```

### 重定位表结构

```
┌────────────────────────────────────────────────────────────┐
│                    重定位表 (.rel.text)                     │
├──────────────┬────────────────┬─────────────┬──────────────┤
│   偏移量      │    类型         │    符号     │    加数      │
├──────────────┼────────────────┼─────────────┼──────────────┤
│   0x00000005 │  R_X86_64_PC32 │  printf(2)  │   -0x4       │
│   0x0000000c │  R_X86_64_32   │  data_var(1)│   +0x0       │
│   0x00000015 │  R_X86_64_PLT32│  malloc(3)  │   -0x4       │
└──────────────┴────────────────┴─────────────┴──────────────┘
```

### 强符号与弱符号

```c
// 强符号 - 多次定义会报错
int global_var = 10;        // 强符号 (已初始化)
void strong_func() { }      // 强符号 (函数定义)

// 弱符号 - 允许重复定义，强符号优先
extern int weak_var __attribute__((weak));
void weak_func() __attribute__((weak));

// 使用场景: 库函数的默认实现
void __attribute__((weak)) platform_init() {
    // 默认空实现
}

// 应用程序可以覆盖
void platform_init() {
    // 自定义硬件初始化
}
```

## 静态链接 vs 动态链接

### 链接方式对比

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           静态链接                                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                               │
│  │ main.o   │  │ math.o   │  │ libc.a   │──┐                            │
│  │ (代码)    │  │ (代码)    │  │ (静态库)  │  │                            │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  │                            │
│       │             │             │        │                            │
│       └─────────────┴─────────────┘        │                            │
│                     │                       │                            │
│                     ▼                       │                            │
│              ┌────────────┐                 │                            │
│              │   链接器    │  复制所需代码    │                            │
│              └─────┬──────┘                 │                            │
│                    │                        │                            │
│                    ▼                        │                            │
│            ┌──────────────┐                 │                            │
│            │   a.out      │◀────────────────┘                            │
│            │ (完整副本)    │                                              │
│            │ 代码: 2MB    │                                              │
│            │ 启动: 快     │                                              │
│            │ 内存: 独占   │                                              │
│            └──────────────┘                                              │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│                           动态链接                                       │
│  ┌──────────┐                                                          │
│  │ main.o   │────┐                                                      │
│  │ (代码)    │    │      ┌──────────────┐     ┌──────────────┐         │
│  └──────────┘    │      │              │     │              │         │
│                  ├─────▶│   a.out      │────▶│  libc.so.6   │         │
│  ┌──────────┐    │      │ (引用记录)    │     │ (共享库)      │         │
│  │ math.o   │────┘      │ 代码: 50KB   │     │ 内存: 共享   │         │
│  └──────────┘           │ 启动: 需解析  │     │ 磁盘: 省空间 │         │
│                         └──────────────┘     └──────────────┘         │
│                                    │                                    │
│                         运行时加载: ld-linux.so                        │
└─────────────────────────────────────────────────────────────────────────┘
```

### 创建和使用静态库

```bash
# 编译目标文件
gcc -c math_utils.c -o math_utils.o
gcc -c string_utils.c -o string_utils.o

# 创建静态库
ar rcs libutils.a math_utils.o string_utils.o

# 查看静态库内容
ar -t libutils.a
nm libutils.a

# 链接静态库
gcc main.c -L. -lutils -o main_static
```

### 创建和使用动态库

```bash
# 编译位置无关代码
gcc -fPIC -c math_utils.c -o math_utils.o

# 创建动态库
gcc -shared -o libutils.so math_utils.o

# 链接动态库
gcc main.c -L. -lutils -o main_dynamic -Wl,-rpath,'$ORIGIN'

# 查看动态库依赖
ldd main_dynamic

# 查看动态符号表
readelf -s libutils.so | grep FUNC
```

### 库搜索顺序

```
编译时搜索路径:
1. -L 指定的路径
2. LIBRARY_PATH 环境变量
3. 系统默认路径 (/lib, /usr/lib, /usr/local/lib)

运行时搜索路径:
1. -Wl,-rpath 编译时指定的路径
2. LD_LIBRARY_PATH 环境变量
3. /etc/ld.so.cache 缓存
4. 系统默认路径 (/lib, /usr/lib)
5. /lib64, /usr/lib64 (64位系统)
```

## 位置无关代码 (PIC) 与位置无关可执行文件 (PIE)

### 为什么需要 PIC/PIE

```
无 PIC/PIE 的问题:
┌─────────────────────────────────────────────────────────┐
│  程序 A (加载地址 0x400000)                              │
│  ┌─────────┐                                            │
│  │ .text   │ 硬编码地址 0x400000 + offset                │
│  └─────────┘                                            │
│                                                         │
│  程序 B (同样需要 0x400000) ← 冲突!                      │
│  ┌─────────┐                                            │
│  │ .text   │ 无法加载，地址已被占用                       │
│  └─────────┘                                            │
└─────────────────────────────────────────────────────────┘

使用 PIC/PIE 的解决方案:
┌─────────────────────────────────────────────────────────┐
│  程序 A (随机加载 ASLR)                                  │
│  ┌─────────┐    ┌─────────┐                             │
│  │ .text   │───▶│  GOT    │───▶ 运行时解析的实际地址      │
│  └─────────┘    └─────────┘                             │
│                                                         │
│  程序 B (不同随机地址)                                   │
│  ┌─────────┐    ┌─────────┐                             │
│  │ .text   │───▶│  GOT    │───▶ 不同的实际地址          │
│  └─────────┘    └─────────┘                             │
│  代码相同，通过 GOT 解引用不同地址                        │
└─────────────────────────────────────────────────────────┘
```

### GOT 和 PLT 机制详解

```
┌───────────────────────────────────────────────────────────────────┐
│                        程序内存布局                                 │
├───────────────────────────────────────────────────────────────────┤
│  .text 代码段                                                     │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │ main:                                                       │  │
│  │   call printf@plt    ──┐                                    │  │
│  │                         │                                    │  │
│  │ 其他代码...             │                                    │  │
│  └────────────────────────┼────────────────────────────────────┘  │
│                           │                                       │
│                           ▼                                       │
│  .plt 过程链接表          ┌─────────────────────────────┐          │
│                           │ printf@plt:                 │          │
│                           │   jmp *printf@got           │──┐       │
│                           │   push $reloc_index         │  │       │
│                           │   jmp .plt0                 │  │       │
│                           └─────────────────────────────┘  │       │
│                                                            │       │
│                                                            ▼       │
│  .got.plt 全局偏移表        ┌─────────────────────────────┐       │
│  (运行时修改)              │ ...                         │       │
│                            │ printf@got: 0x4004a6 (plt) │◀──┘ 第一次
│                            │    ↓ 动态链接器解析后        │          │
│                            │ printf@got: 0x7f... (真实)  │◀──┘ 后续
│                            └─────────────────────────────┘          │
│                                                                     │
│  .got 全局偏移表 (数据)                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ global_var@got: 0x7f... (实际地址)                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────────────────┘
```

### PIC 代码示例

```c
// 非 PIC 代码 (x86-64)
// mov global_var(%rip), %eax  直接访问

// PIC 代码 - 通过 GOT 访问全局变量
// 编译: gcc -fPIC -S test.c

static int local_var = 10;  // 内部符号，不需要 GOT
extern int shared_var;       // 外部符号，需要 GOT

int get_shared() {
    return shared_var;       // 通过 GOT 访问
}

// 生成的汇编 (简化):
// get_shared:
//     movq shared_var@GOTPCREL(%rip), %rax  # 获取 GOT 条目地址
//     movl (%rax), %eax                      # 解引用获取值
//     ret
```

### PIE 编译选项

```bash
# 完全位置无关的可执行文件
gcc -fPIE -pie main.c -o main_pie

# 检查是否为 PIE
readelf -h main_pie | grep Type
# Type: DYN (Shared object file)  ← 这是 PIE 可执行文件

# 检查 ASLR 状态
cat /proc/sys/kernel/randomize_va_space
# 0 = 关闭, 1 = 保守, 2 = 完全随机

# 查看加载地址
pmap $(pidof main_pie)
```

## 链接时优化 (LTO)

### LTO 的工作原理

```
传统编译模型:
┌────────────┐    ┌────────────┐    ┌────────────┐
│  file1.c   │───▶│  file1.o   │───┐
└────────────┘    │ (机器码)    │   │
                  └────────────┘   │    ┌────────────┐
                                   ├───▶│   链接器    │───▶ a.out
┌────────────┐    ┌────────────┐   │    │ (无法优化)  │
│  file2.c   │───▶│  file2.o   │───┘    └────────────┘
└────────────┘    │ (机器码)    │
                  └────────────┘

LTO 编译模型:
┌────────────┐    ┌──────────────────┐
│  file1.c   │───▶│   file1.o (IR)   │──┐
└────────────┘    │ (LLVM/GCC 中间码) │  │
                  └──────────────────┘  │   ┌────────────────────────┐
                                        ├──▶│    链接器 = 编译器      │
┌────────────┐    ┌──────────────────┐  │   │  - 跨模块内联           │
│  file2.c   │───▶│   file2.o (IR)   │──┘   │  - 全局值编号          │
└────────────┘    │ (LLVM/GCC 中间码) │      │  - 死代码消除          │
                  └──────────────────┘      │  - 全局优化             │
                                            └───────────┬────────────┘
                                                        │
                                                        ▼
                                                  ┌────────────┐
                                                  │   a.out    │
                                                  │ (优化后机器码)│
                                                  └────────────┘
```

### 启用 LTO

```bash
# GCC LTO
gcc -flto -O2 -c file1.c -o file1.o
gcc -flto -O2 -c file2.c -o file2.o
gcc -flto -O2 file1.o file2.o -o program

# 或者一步完成
gcc -flto -O2 file1.c file2.c -o program

# Clang LTO (ThinLTO - 更快，内存效率更高)
clang -flto=thin -O2 -c file1.c -o file1.o
clang -flto=thin -O2 -c file2.c -o file2.o
clang -flto=thin -O2 file1.o file2.o -o program

# 使用链接器插件 (Gold/LLD)
gcc -flto -O2 -fuse-linker-plugin file1.o file2.o -o program
```

### LTO 优化效果示例

```c
// math_lib.c
int add(int a, int b) {
    return a + b;
}

// main.c
#include <stdio.h>

int add(int a, int b);  // 声明

int main() {
    int result = add(3, 4);  // 可以被内联
    printf("%d\n", result);
    return 0;
}
```

```bash
# 传统编译 - add 作为独立函数调用
gcc -O2 -c math_lib.c -o math_lib.o
gcc -O2 -c main.c -o main.o
gcc math_lib.o main.o -o no_lto
objdump -d no_lto | grep -A5 "call.*add"

# LTO 编译 - add 被内联，无函数调用开销
gcc -flto -O2 math_lib.c main.c -o with_lto
objdump -d with_lto | grep "call.*add"  # 无输出，已内联
```

### LTO 的权衡

| 优势 | 劣势 |
|------|------|
| 跨模块内联 | 编译时间显著增加 |
| 更激进的优化 | 内存消耗增加 |
| 更好的死代码消除 | 调试信息可能不准确 |
| 全局常量传播 | 增量编译效果差 |
| 更好的向量化 | 链接器需支持插件 |

## 调试链接问题

### 常见链接错误及解决

```bash
# 1. 未定义引用 (undefined reference)
# 错误: undefined reference to `sqrt'
# 解决: 添加 -lm 链接数学库
gcc main.c -o main -lm

# 2. 多重定义 (multiple definition)
# 错误: multiple definition of `global_var'
# 解决: 头文件中使用 extern，只有一个源文件定义

# 3. 符号版本不匹配
# 使用 objdump 查看符号版本
objdump -T /lib/x86_64-linux-gnu/libc.so.6 | grep GLIBC

# 4. 库依赖问题
# 查看完整依赖树
ldd -v ./program

# 5. 动态链接器无法找到库
# 临时方案
export LD_LIBRARY_PATH=/path/to/lib:$LD_LIBRARY_PATH
# 永久方案
sudo ldconfig -n /path/to/lib
```

## 总结

链接和加载是程序构建过程中的关键环节，理解其工作原理有助于：

1. **优化构建时间** - 合理使用增量链接、预编译头
2. **减小二进制体积** - 使用动态链接、LTO 消除死代码
3. **提升运行时性能** - PIC/PIE 启用 ASLR，延迟绑定减少启动时间
4. **解决复杂问题** - 符号冲突、版本兼容性、加载失败

掌握这些底层知识，能够更好地理解程序的执行模型，并在遇到链接错误时快速定位和解决问题。
