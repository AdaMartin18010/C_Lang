## 1. 嵌入式内存布局形式化定义

### 1.1 内存映射层次结构

```

│              嵌入式系统内存映射 (典型ARM Cortex-M)                │

│                                                                 │
│  高地址                                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  栈 (Stack)               ↑ 向下增长                     │   │
│  │  - 局部变量                                              │   │

│  │  - 返回地址                                              │   │

│  低地址 (0x0000_0000 或 0x0800_0000)                             │
│                                                                 │
│  Flash/ROM:                                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  中断向量表                                              │   │
│  │  .text (代码)                                           │   │
│  │  .rodata (常量)                                         │   │
│  │  启动代码 (Reset_Handler)                               │   │

│                                                                 │
```

### 1.2 内存区域属性矩阵

| 区域 | 位置 | 生命周期 | 初始化 | 访问速度 | 大小限制 | 确定性 |
|------|------|----------|--------|----------|----------|--------|
| **栈** | RAM | 函数作用域 | 自动 | 快 | 小(几KB) | 高 |
| **堆** | RAM | 程序员控制 | 手动 | 快 | 可变 | 低(碎片) |
| **.data** | RAM | 程序运行期 | 启动时 | 快 | 中 | 高 |
| **.bss** | RAM | 程序运行期 | 启动时清零 | 快 | 中 | 高 |
| **.text** | Flash | 程序运行期 | 编译时 | 慢 | 大 | 高 |
| **.rodata** | Flash | 程序运行期 | 编译时 | 慢 | 中 | 高 |

### 1.3 形式化内存模型

**定义 1.1 (嵌入式内存空间)**

```
嵌入式内存空间 M = (RAM ∪ Flash ∪ ROM)

RAM = Stack ∪ Heap ∪ DataSegment

其中:
  Stack = {addr | StackBase ≥ addr ≥ StackLimit}
  Heap = {addr | HeapBase ≤ addr ≤ HeapLimit}
  DataSegment = .data ∪ .bss

约束:
  1. Stack ∩ Heap = ∅ (栈堆不重叠，或需碰撞检测)
  2. StackGrowthDirection = DOWN (向低地址增长)
  3. HeapGrowthDirection = UP (向高地址增长)
  4. |Stack| << |RAM| (栈通常只占RAM的一小部分)
```

## 2. 栈管理深度分析

### 2.1 栈溢出检测机制

**定义 2.1 (栈溢出)**

```
栈溢出条件:
  SP < StackLimit

其中:
  SP = 当前栈指针值
  StackLimit = 栈底边界地址

后果:
  1. 写入堆或其他数据区 → 数据损坏
  2. 写入无效地址 → 总线错误/HardFault
  3. 无MMU系统无法捕获 → 静默损坏
```

**栈溢出检测决策树**

```
栈溢出检测策略
│
├── 硬件检测 (如果支持)
│   ├── MPU (Memory Protection Unit)
│   │   └── 配置栈区域为不可写
   ├── 栈限制寄存器 (如ARMv8-M)
│   │   └── 硬件自动检查SP
│   └── 优点: 零开销，立即检测
│
├── 软件检测
│   ├── 栈填充图案 (Stack Canaries)
│   │   └── 在栈底填充已知图案(如0xDEADBEEF)
│   │   └── 定期检查图案完整性
│   │   └── 开销: 周期性检查
│   │
│   ├── 栈水印 (Stack Watermark)
│   │   └── 初始化时填充整个栈区域
│   │   └── 运行时检查最低水位
│   │   └── 用于统计最大栈使用量
│   │
│   └── 运行时检查
│       └── 函数入口检查SP
│       └── 开销大，很少使用
│
└── 静态分析
    └── 编译时计算栈深度
    └── 限制: 无法处理递归和动态调用
```

### 2.2 栈大小估算方法

**定理 2.1 (最大栈深度计算)**

```
最大栈深度 = 中断嵌套深度 × (中断上下文大小 + 中断处理代码栈使用)
           + 最大调用链深度 × 平均函数栈帧
           + 局部变量峰值

其中:
  中断上下文大小 = 保存的寄存器数量 × 4字节

示例 (ARM Cortex-M4):
  - 硬件保存: R0-R3, R12, LR, PC, xPSR = 8×4 = 32字节
  - 软件保存: R4-R11 = 8×4 = 32字节 (如果需要)
  - 中断处理: 平均50字节
  - 最大嵌套: 3级 (考虑优先级)
  - 中断栈: 3 × (32 + 50) = 246字节

  - 应用调用链: 10层深度
  - 每层栈帧: 平均20字节
  - 应用栈: 10 × 20 = 200字节

  - 安全余量: 25%
  - 总栈大小: (246 + 200) × 1.25 ≈ 560字节
  - 实际配置: 1KB (留有余量)
```

## 3. 堆管理策略

### 3.1 嵌入式堆使用决策

```
嵌入式系统中是否使用堆?
│
├── 严格实时/安全关键系统 (汽车、航空)
│   └── 禁止使用动态分配
│       ├── 原因1: 分配时间不确定 (碎片整理)
│       ├── 原因2: 可能分配失败 (内存不足)
│       ├── 原因3: 碎片化导致长期问题
│       └── 替代方案: 静态分配、内存池
│
├── 中等复杂度系统
│   └── 受限使用
│       ├── 初始化时分配 (运行期不释放)
│       ├── 固定大小块分配 (避免碎片)
│       └── 监控堆使用情况
│
└── 复杂系统 (Linux级别)
    └── 完整堆管理
        ├── 使用高级分配器 (TLSF等)
        ├── 确定性分配算法
        └── 内存压缩/整理
```

### 3.2 内存分配算法对比矩阵

| 算法 | 分配复杂度 | 释放复杂度 | 碎片 | 确定性 | 嵌入式适用 |
|------|------------|------------|------|--------|------------|
| **首次适应** | O(n) | O(1) | 中 | 否 | ⭐⭐ |
| **最佳适应** | O(n) | O(1) | 高 | 否 | ⭐ |
| **最差适应** | O(n) | O(1) | 低 | 否 | ⭐⭐ |
| **伙伴系统** | O(log n) | O(log n) | 中 | 部分 | ⭐⭐⭐ |
| **TLSF** | O(1) | O(1) | 低 | **是** | ⭐⭐⭐⭐⭐ |
| **内存池** | O(1) | O(1) | 无 | **是** | ⭐⭐⭐⭐⭐ |

### 3.3 内存池实现

```c
/* 内存池形式化定义 */

/* 内存池块大小等级 */
#define POOL_BLOCK_SIZES  {16, 32, 64, 128, 256, 512, 1024}
#define POOL_NUM_CLASSES  7

/* 内存池块头 */
typedef struct PoolBlock {
    struct PoolBlock *next;     /* 下一个空闲块 */
    uint8_t class_index;        /* 大小等级 */
    uint8_t in_use;             /* 使用标志 */
} PoolBlock;

/* 内存池类 */
typedef struct PoolClass {
    size_t block_size;          /* 块大小 */
    size_t num_blocks;          /* 块数量 */
    PoolBlock *free_list;       /* 空闲链表 */
    uint8_t *pool_memory;       /* 池内存 */
} PoolClass;

/* 内存池管理器 */
typedef struct MemoryPool {
    PoolClass classes[POOL_NUM_CLASSES];
    size_t total_size;
    size_t used_size;
    uint32_t alloc_count;
    uint32_t free_count;
} MemoryPool;

/* 分配算法: O(1) 时间复杂度 */
void *pool_alloc(MemoryPool *pool, size_t size) {
    /* 1. 找到合适的大小等级 */
    int class_idx = find_size_class(size);
    if (class_idx < 0 || class_idx >= POOL_NUM_CLASSES) {
        return NULL;  /* 请求过大 */
    }

    PoolClass *pc = &pool->classes[class_idx];

    /* 2. 检查空闲链表 */
    if (pc->free_list == NULL) {
        return NULL;  /* 该等级池耗尽 */
    }

    /* 3. 从链表头取出块 */
    PoolBlock *block = pc->free_list;
    pc->free_list = block->next;
    block->in_use = 1;

    /* 4. 更新统计 */
    pool->used_size += pc->block_size;
    pool->alloc_count++;

    /* 5. 返回数据区指针 (跳过头) */
    return (void *)(block + 1);
}

/* 释放算法: O(1) 时间复杂度 */
void pool_free(MemoryPool *pool, void *ptr) {
    if (ptr == NULL) return;

    /* 1. 获取块头 */
    PoolBlock *block = (PoolBlock *)ptr - 1;

    /* 2. 检查有效性 */
    if (!block->in_use) return;  /* 重复释放 */

    int class_idx = block->class_index;
    PoolClass *pc = &pool->classes[class_idx];

    /* 3. 插入空闲链表头部 */
    block->next = pc->free_list;
    pc->free_list = block;
    block->in_use = 0;

    /* 4. 更新统计 */
    pool->used_size -= pc->block_size;
    pool->free_count++;
}

/* 内存池优势证明:
 * 1. 分配时间: O(1) - 确定性的实时系统友好
 * 2. 无外部碎片 - 固定块大小
 * 3. 无分配失败 (只要池有空间)
 * 4. 内存对齐 - 自动处理
 */
```

## 4. 内存对齐与性能

### 4.1 对齐要求矩阵

| 数据类型 | 大小 | 对齐要求 | 未对齐访问后果 |
|----------|------|----------|----------------|
| **uint8_t** | 1 | 1 | 无 |
| **uint16_t** | 2 | 2 | 可能变慢或异常 |
| **uint32_t** | 4 | 4 | 可能变慢或异常 |
| **uint64_t** | 8 | 8 | 可能异常 (ARMv7-M) |
| **float** | 4 | 4 | 性能下降 |
| **double** | 8 | 8 | 性能下降或异常 |
| **指针** | 4/8 | 4/8 | 可能异常 |

### 4.2 栈对齐定理

**定理 4.1 (AAPCS栈对齐要求)**

```
定理: ARM架构过程调用标准(AAPCS)要求栈8字节对齐

证明:
  1. 保证64位数据(double, long long)正确访问
  2. 保证SIMD指令(如VSTM)正确执行
  3. 保证外部库接口兼容

实现:
  - 函数入口: SP = (SP - local_size) & ~0x7
  - 中断保存: 自动处理对齐

违规后果:
  - 未对齐访问导致UsageFault (如果使能)
  - 或性能显著下降
```

### 5.2 内存监控实现

```c
/* 内存使用监控 */

/* 栈监控 */
typedef struct {
    uint32_t *stack_bottom;
    uint32_t *stack_top;
    uint32_t watermark;         /* 最高使用位置 */
    uint32_t pattern;           /* 填充图案 */
} StackMonitor;

void stack_monitor_init(StackMonitor *sm, uint32_t *stack_base, size_t size) {
    sm->stack_bottom = stack_base;
    sm->stack_top = stack_base + size / 4;
    sm->watermark = sm->stack_top;
    sm->pattern = 0xDEADBEEF;

    /* 填充栈区域 */
    for (uint32_t *p = sm->stack_bottom; p < sm->stack_top; p++) {
        *p = sm->pattern;
    }
}

size_t stack_get_free(StackMonitor *sm) {
    /* 查找最低的非图案值 */
    uint32_t *p = sm->stack_bottom;
    while (p < sm->watermark && *p == sm->pattern) {
        p++;
    }
    sm->watermark = p;
    return (sm->stack_top - p) * 4;
}

size_t stack_get_used(StackMonitor *sm) {
    return (sm->stack_top - sm->stack_bottom) * 4 - stack_get_free(sm);
}

uint8_t stack_check_overflow(StackMonitor *sm) {
    return (sm->stack_bottom[0] != sm->pattern);
}
```
