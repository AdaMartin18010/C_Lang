### 2.1 技术维度评分

| 维度 | 权重 | C语言 | Rust | 说明 |

| **编译速度** | 10% | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | C编译明显更快 |
| **调试便利性** | 10% | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | GDB对C支持更成熟 |
| **平台支持** | 5% | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | C支持几乎所有平台 |

## 4. 风险评估框架

### 4.1 技术风险矩阵

| 风险项 | C语言 | Rust | 缓解措施 |
|:-------|:-----:|:----:|:---------|
| **内存安全漏洞** | 高 | 低 | Rust: 借用检查器；C: 使用MISRA+静态分析 |
| **并发Bug** | 高 | 低 | Rust: Send/Sync trait；C: 严格锁规范 |
| **开发延迟** | 低 | 中 | Rust: 提前培训；预留学习时间 |
| **招聘困难** | 低 | 中 | Rust: 内部培养；提供培训预算 |
| **工具链不成熟** | 低 | 中 | Rust: 关注内核/嵌入式支持状态 |
| **供应商锁定** | 中 | 中 | 两者: 保持模块边界清晰 |

### 4.2 决策风险检查

```
选择Rust前确认：
☐ 团队至少有1名Rust熟练开发者
☐ 项目时间线允许2-4周学习曲线
☐ 目标平台有Tier 1/2 Rust支持
☐ 关键依赖库有Rust绑定或替代方案
☐ 有明确的Rust编码规范
☐ CI/CD支持Rust工具链

选择C前确认：
☐ 有成熟的静态分析工具链（Coverity等）
☐ 团队遵循MISRA/CERT C编码规范
☐ 定期进行模糊测试和安全审计
☐ 关键路径有形式化验证覆盖
☐ 内存安全漏洞响应流程就位
```

### 5.2 模块优先级评估

```python
# 模块优先级计算示例
def calculate_priority(module):
    """
    评分越高，越优先迁移到Rust
    """
    score = 0

    # 安全风险（权重：40%）
    score += module.cve_history.count() * 10  # 每个历史CVE +10
    score += module.handles_untrusted_input * 30  # 处理外部输入 +30
    score += module.network_facing * 25  # 网络边界 +25

    # 业务价值（权重：30%）
    score += module.criticality * 10  # 关键程度 1-10

    # 技术可行性（权重：30%）
    score -= module.dependencies_count * 2  # 依赖越多越难迁移
    score += module.test_coverage / 10  # 测试覆盖率高更容易验证
    score -= module.lines_of_code / 1000  # 代码量大的模块后移

    return score

# 优先级分级
# score > 80:  立即迁移（P0）
# score 50-80: 短期迁移（P1）
# score 20-50: 中期迁移（P2）
# score < 20:  评估保留（P3）
```

### 6.2 ROI计算示例

```
场景：安全关键系统，历史每年10个CVE
─────────────────────────────────────────────────────────────

迁移成本：$200K（一次性）
投资回收期：3.5个月

5年ROI：($690K × 5 - $200K) / $200K = 1625%
```
