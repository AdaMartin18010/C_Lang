## 第二部分：malloc/free深入 (4:00 - 9:00)

### 4:00 | malloc基础 | 画面：malloc函数的原型和基本用法

**讲解词**:
malloc是C语言最核心的内存分配函数。它从堆中分配指定字节数的内存，返回指向该内存的指针。如果分配失败，返回NULL。

**代码演示**:

```c
#include <stdlib.h>

// 基本用法
int *arr = malloc(100 * sizeof(int));
if (arr == NULL) {
    // 处理分配失败
    perror("malloc failed");
    exit(1);
}

// 使用内存
for (int i = 0; i < 100; i++) {
    arr[i] = i;
}

// 释放内存
free(arr);
arr = NULL;  // 防止野指针
```

calloc在分配内存的同时将其初始化为零，适合分配数组。realloc可以调整已分配内存块的大小，可能涉及数据复制。

### 6:30 | 内存碎片 | 画面：内部碎片和外部碎片的可视化

**讲解词**:
内存碎片是动态分配的固有问题。内部碎片是分配的内存大于请求的大小造成的浪费；外部碎片是空闲内存被分割成小块，无法满足大的分配请求。

**碎片类型可视化**:

```
外部碎片问题:
┌──────┬──────┬──────┬──────┬──────┬──────┐
│ 空闲  │ 已用  │ 空闲  │ 已用  │ 空闲  │ 已用  │
└──────┴──────┴──────┴──────┴──────┴──────┘
 64B    64B    64B    64B    64B    64B

总空闲: 192B，但无法分配 128B 的连续块！

内部碎片:
请求: 100字节，分配: 128字节 (对齐)
┌─────────────────────────────────────┐
│ 100字节数据 │ 28字节浪费 (内部碎片)  │
└─────────────────────────────────────┘
```

### 7:30 | 内存分配策略 | 画面：首次适应、最佳适应、最差适应算法的对比

**讲解词**:
分配器使用不同策略选择空闲块。首次适应找到第一个足够大的块；最佳适应找最小的满足条件的块，减少浪费；最差适应使用最大的块，保留大空闲区。

**算法对比**:

```
空闲块链表: [16B] → [64B] → [128B] → [256B] → NULL

malloc(100):
├── 首次适应: 选择 [128B] 块
├── 最佳适应: 选择 [128B] 块 (最小满足)
└── 最差适应: 选择 [256B] 块 (最大块)

碎片影响:
├── 首次适应: 低地址聚集碎片
├── 最佳适应: 产生大量小碎片
└── 最差适应: 碎片分布较均匀
```

### 8:15 | 分配器演进 | 画面：ptmalloc、jemalloc、tc malloc的对比

**讲解词**:
Linux系统默认使用ptmalloc。对于高并发场景，可以使用jemalloc或tcmalloc，它们使用线程本地缓存减少锁竞争，提供更好的多核性能。

**分配器对比**:

| 分配器 | 特点 | 适用场景 |
|-------|------|---------|
| ptmalloc | 通用，兼容性好 | 一般应用 |
| jemalloc | 多核友好，低碎片 | 数据库、缓存 |
| tcmalloc | 线程缓存，高性能 | 高并发服务 |
| mimalloc | 小对象优化 | 游戏引擎 |

## 第三部分：常见内存错误 (9:00 - 14:00)

### 9:00 | 内存泄漏 | 画面：泄漏检测工具的输出，内存使用曲线

**讲解词**:
内存泄漏是最常见的内存错误——分配的内存没有释放。短期程序可能不受影响，但长期运行的服务会逐渐耗尽内存。检测工具如Valgrind可以帮助定位泄漏。

**泄漏模式可视化**:

```c
// 泄漏示例1: 丢失指针
void leak1() {
    int *p = malloc(100);
    p = malloc(200);  // 第一个100字节丢失！
    free(p);
}

// 泄漏示例2: 异常路径未释放
void leak2() {
    int *p = malloc(100);
    if (some_error()) {
        return;  // 内存泄漏！
    }
    free(p);
}

// 泄漏示例3: 数据结构中的循环引用
// 需要更复杂的检测和打破循环
```

### 9:50 | 使用已释放内存 | 画面：Use-after-free攻击的简化演示

**讲解词**:
使用已释放的内存称为use-after-free，是严重的安全漏洞。攻击者可能利用这一点执行任意代码。释放后立即将指针设为NULL是良好的防御习惯。

**错误演示**:

```c
void use_after_free() {
    int *p = malloc(sizeof(int));
    *p = 42;
    free(p);

    // 危险！p现在是悬空指针
    printf("%d\n", *p);  // Use-after-free! 未定义行为

    // 正确做法
    p = NULL;  // 释放后置空
}
```

### 10:35 | 双重释放 | 画面：堆损坏的连锁反应

**讲解词**:
双重释放同一块内存会导致堆数据结构损坏，可能引发崩溃或安全漏洞。释放后置NULL可以避免双重释放，因为对NULL调用free是安全的。

**双重释放危害**:

```c
void double_free() {
    int *p = malloc(100);
    free(p);
    free(p);  // 双重释放！堆损坏
}

// 安全模式
void safe_free(void **pp) {
    if (pp != NULL && *pp != NULL) {
        free(*pp);
        *pp = NULL;
    }
}

// 使用
safe_free((void**)&p);  // 第一次释放
safe_free((void**)&p);  // 安全，无操作
```

**讲解词**:
缓冲区溢出发生在写入超出分配内存边界时。栈缓冲区溢出可能被利用来覆盖返回地址，执行恶意代码。这是最常见的安全漏洞类型之一。

溢出后:
┌──────────────────┐

├──────────────────┤

### 12:15 | 未初始化内存 | 画面：未初始化值的随机性展示

**讲解词**:
使用未初始化的内存会导致未定义行为——程序可能偶尔正确，偶尔崩溃，难以调试。malloc不初始化内存，calloc会清零，栈上的变量值不确定。

**代码示例**:

```c
void uninitialized() {
    int x;           // 未初始化！值不确定
    printf("%d\n", x);  // 未定义行为

    // 正确做法
    int y = 0;       // 明确初始化
    int *p = calloc(1, sizeof(int));  // calloc清零

    free(p);
}
```

### 12:50 | 内存对齐问题 | 画面：未对齐访问的性能影响和错误

**讲解词**:
某些架构要求数据按特定边界对齐。未对齐访问可能导致性能下降或硬件异常。malloc返回的内存总是适当对齐的，但指针运算可能破坏对齐。

地址:  0x1000  0x1001  0x1002  0x1003  0x1004
       ┌───────┬───────┬───────┬───────┬───────┐

### 19:00 | 为什么需要自定义分配器 | 画面：通用分配器的局限性和场景需求

**讲解词**:
通用分配器为了适应各种场景，会做一些权衡。在特定场景下，自定义分配器可以显著改善性能和内存使用。比如游戏引擎中的对象池、数据库中的固定大小分配。

**场景分析**:

```
通用malloc的问题:
├── 锁竞争 (多线程)
├── 内存碎片
├── 元数据开销 (8-16字节/块)
└── 分配/释放速度

自定义分配器的优势:
├── 无锁设计 (每线程池)
├── 特定分配模式优化
├── 零元数据开销
└── O(1) 分配/释放
```

### 19:45 | 简单内存池设计 | 画面：内存池的数据结构和算法

**讲解词**:
最简单的自定义分配器是内存池。预先分配一大块内存，然后从中切分小块。适用于分配大量相同大小对象的场景。

**内存池实现**:

```c
typedef struct PoolBlock {
    struct PoolBlock *next;
} PoolBlock;

typedef struct {
    size_t block_size;
    size_t blocks_per_chunk;
    PoolBlock *free_list;
    void *chunks;  // 大块内存列表
} MemoryPool;

// 初始化内存池
void pool_init(MemoryPool *pool, size_t block_size, size_t blocks) {
    pool->block_size = block_size < sizeof(PoolBlock)
                       ? sizeof(PoolBlock) : block_size;
    pool->blocks_per_chunk = blocks;
    pool->free_list = NULL;
    pool->chunks = NULL;
}
```

### 20:40 | 内存池分配与释放 | 画面：分配算法的可视化

**讲解词**:
内存池的分配非常简单——从空闲链表取一个块。释放时把块放回链表。都是O(1)操作，比通用malloc快得多。

**分配过程动画**:

```
空闲链表初始状态:
free_list ──► [块1] ──► [块2] ──► [块3] ──► NULL

pool_alloc():
1. 取free_list头部
2. free_list = free_list->next
3. 返回原头部

分配后:
free_list ──► [块2] ──► [块3] ──► NULL
返回: [块1]

pool_free(块1):
1. 块1->next = free_list
2. free_list = 块1

释放后:
free_list ──► [块1] ──► [块2] ──► [块3] ──► NULL
```

### 21:30 | 完整内存池实现 | 画面：完整代码展示和讲解

**讲解词**:
让我们看一个完整的内存池实现。当空闲链表为空时，我们分配一个新的大块内存，将其切分成小块加入链表。

**完整代码**:

```c
void *pool_alloc(MemoryPool *pool) {
    // 空闲链表为空，分配新的大块
    if (pool->free_list == NULL) {
        size_t chunk_size = pool->block_size * pool->blocks_per_chunk;
        char *chunk = malloc(chunk_size);
        if (chunk == NULL) return NULL;

        // 将新大块加入chunks列表
        *(void**)chunk = pool->chunks;
        pool->chunks = chunk;

        // 切分并加入空闲链表
        for (size_t i = 0; i < pool->blocks_per_chunk; i++) {
            PoolBlock *block = (PoolBlock*)(chunk + i * pool->block_size);
            block->next = pool->free_list;
            pool->free_list = block;
        }
    }

    // 从空闲链表取一个块
    PoolBlock *block = pool->free_list;
    pool->free_list = block->next;
    return block;
}

void pool_free(MemoryPool *pool, void *ptr) {
    if (ptr == NULL) return;
    PoolBlock *block = ptr;
    block->next = pool->free_list;
    pool->free_list = block;
}
```
