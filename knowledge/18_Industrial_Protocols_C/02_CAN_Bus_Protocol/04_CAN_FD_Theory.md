## 1. CAN FD 核心创新形式化定义

### 1.1 CAN 2.0 vs CAN FD 差异矩阵

| 特性 | CAN 2.0 | CAN FD | 改进倍数 | C语言影响 |
|------|---------|--------|----------|-----------|
| **数据长度** | 0-8字节 | 0-64字节 | 8x | 缓冲区增大 |
| **数据段速率** | ≤1Mbps | ≤8Mbps | 8x | 时钟精度要求↑ |
| **CRC长度** | 15位 | 17/21位 | 1.4x | CRC计算变化 |
| **填充位** | 每5位 | 每4位 (数据段) | - | 位填充算法变化 |
| **远程帧** | 支持 | 不支持 | - | 简化处理 |
| **帧格式** | 固定 | 可变速率 | - | 控制器配置复杂 |

## 2. DLC 编码形式化定义

### 2.1 数据长度码映射

**定理 2.1 (DLC编码非线性)**

```
CAN FD 的 DLC 编码不是简单的线性映射，超过8字节后采用非线性编码:

定义: DLC ∈ {0, 1, 2, ..., 15}

映射函数:
  DataLength(dlc) =
    dlc                    , if dlc ≤ 8
    12 + (dlc - 9) × 4     , if dlc > 8

具体映射表:
┌─────┬──────────────┐
│ DLC │ 数据字节数   │
├─────┼──────────────┤
│ 0-8 │ 0-8          │  (线性)
│  9  │ 12           │
│ 10  │ 16           │
│ 11  │ 20           │
│ 12  │ 24           │
│ 13  │ 32           │
│ 14  │ 48           │
│ 15  │ 64           │
└─────┴──────────────┘

证明这种编码的合理性:
1. 向后兼容: dlc=0-8与CAN 2.0相同
2. 粒度优化: 小数据用细粒度(4字节步进), 大数据用粗粒度
3. 编码效率: 4位DLC可表示16种长度，覆盖0-64字节
```

### 2.2 C语言实现

```c
/* DLC 到数据长度转换 */
static const uint8_t dlc_to_length[] = {
    0, 1, 2, 3, 4, 5, 6, 7, 8,    /* DLC 0-8: 线性 */
    12, 16, 20, 24, 32, 48, 64    /* DLC 9-15: 非线性 */
};

uint8_t can_fd_dlc_to_length(uint8_t dlc) {
    if (dlc > 15) {
        return 0;  /* 非法DLC */
    }
    return dlc_to_length[dlc];
}

/* 数据长度到DLC转换 (向上取整) */
uint8_t can_fd_length_to_dlc(uint8_t length) {
    if (length <= 8) {
        return length;
    } else if (length <= 12) {
        return 9;
    } else if (length <= 16) {
        return 10;
    } else if (length <= 20) {
        return 11;
    } else if (length <= 24) {
        return 12;
    } else if (length <= 32) {
        return 13;
    } else if (length <= 48) {
        return 14;
    } else if (length <= 64) {
        return 15;
    } else {
        return 15;  /* 超过64字节按64处理 */
    }
}
```

## 3. CRC 算法递归定义

### 3.1 CRC17 和 CRC21 形式化定义

**定义 3.1 (CAN FD CRC)**

```
CAN FD 使用两种CRC长度，根据数据长度自动选择:

CRC长度选择:
  CRC_Length(data_len) =
    17, if data_len ≤ 16 bytes
    21, if data_len > 16 bytes

生成多项式:
  CRC17: G(x) = x¹⁷ + x¹⁶ + x¹⁴ + x¹³ + x¹¹ + x⁶ + x⁴ + x¹ + 1
           = 0x1685B (0b1 0110 1000 0101 1011)

  CRC21: G(x) = x²¹ + x²⁰ + x¹³ + x¹¹ + x⁷ + x⁴ + x³ + x¹ + 1
           = 0x102ACF (0b1 0000 0010 1010 1100 1111)

注意: CAN FD CRC计算包含填充位计数器，防止填充位错误未检测
```

**定理 3.1 (CRC填充位计数器)**

```
问题: 标准CRC无法检测填充位错误(位流中插入或删除填充位)

解决方案: CRC计算前加入填充位计数器字段

填充位计数器 (Stuff Bit Counter, SBC):
  - 3位格雷码计数值
  - 1位奇偶校验

位置: 位于数据段之后，CRC序列之前

证明这种设计的有效性:
1. 格雷码: 相邻数值只有1位变化，减少错误传播
2. 奇偶校验: 检测SBC本身的错误
3. 计数器范围: 每8位数据最多1个填充位，3位可计数0-7
```

### 3.2 CRC递归算法实现

```c
/* CAN FD CRC17 计算 */
uint32_t can_fd_crc17(const uint8_t *data, size_t len,
                      uint8_t stuff_count) {
    const uint32_t polynomial = 0x1685B;
    uint32_t crc = 0x00000;  /* 初始值 */

    /* 添加填充位计数器 (4位: 3位格雷码 + 1位奇偶校验) */
    uint8_t sbc_with_parity = stuff_count; /* 假设已包含奇偶校验 */

    /* 处理填充位计数器 */
    for (int i = 3; i >= 0; i--) {
        uint8_t bit = (sbc_with_parity >> i) & 1;
        crc = (crc << 1) ^ ((crc >> 16) ^ bit ? polynomial : 0);
        crc &= 0x1FFFF;  /* 17位掩码 */
    }

    /* 处理数据 */
    for (size_t i = 0; i < len; i++) {
        for (int j = 7; j >= 0; j--) {
            uint8_t bit = (data[i] >> j) & 1;
            crc = (crc << 1) ^ ((crc >> 16) ^ bit ? polynomial : 0);
            crc &= 0x1FFFF;
        }
    }

    return crc;
}

/* CAN FD CRC21 计算 */
uint32_t can_fd_crc21(const uint8_t *data, size_t len,
                      uint8_t stuff_count) {
    const uint32_t polynomial = 0x102ACF;
    uint32_t crc = 0x000000;  /* 初始值 */

    /* 类似CRC17的处理，使用21位多项式 */
    /* ... */

    return crc & 0x1FFFFF;  /* 21位掩码 */
}
```

## 4. 位填充算法差异

### 4.1 填充规则对比

| 特性 | CAN 2.0 | CAN FD (仲裁段) | CAN FD (数据段) |
|------|---------|-----------------|-----------------|
| **填充位** | 每5个连续相同位 | 每5个连续相同位 | 每4个连续相同位 |
| **填充位置** | 固定格式位除外 | 固定格式位除外 | 固定格式位除外 |
| **CRC字段** | 填充 | 填充 | **不填充** |
| **stuffed bits** | 最多24个 | 最多24个 | 最多更多 |

## 5. CAN FD 与 CAN 2.0 兼容性分析

### 5.1 兼容性决策树

```
CAN FD 网络设计决策
│
├── 网络是否全为CAN FD节点?
│   ├── YES → 纯CAN FD网络
│   │           ├── 数据段速率: 最高8Mbps
│   │           └── 数据长度: 最高64字节
│   │
│   └── NO → 混合网络 (CAN FD + CAN 2.0)
│       │
│       ├── CAN FD节点发送给CAN 2.0节点?
│       │   ├── YES → 必须限制:
│       │   │           ├── DLC ≤ 8
│       │   │           └── 数据段速率 = 仲裁段速率 (BRS=0)
│       │   │       (否则CAN 2.0节点会报错帧)
│       │   │
│       │   └── NO → 可使用CAN FD特性
│       │
│       └── CAN 2.0节点发送?
│           └── CAN FD节点可正常接收 (向后兼容)
│
└── 升级策略
    ├── 阶段1: 添加CAN FD节点，以CAN 2.0模式运行
    ├── 阶段2: 逐步替换所有节点为CAN FD
    └── 阶段3: 启用CAN FD高速特性
```

### 5.2 混用网络约束矩阵

| 发送节点 | 接收节点 | 数据长度 | 数据段速率 | 结果 |
|----------|----------|----------|------------|------|
| CAN 2.0 | CAN 2.0 | ≤8字节 | 1Mbps | ✅ 正常 |
| CAN 2.0 | CAN FD | ≤8字节 | 1Mbps | ✅ 正常 |
| CAN FD | CAN FD | ≤64字节 | 8Mbps | ✅ 正常 |
| CAN FD | CAN FD | ≤8字节 | 1Mbps | ✅ 正常 |
| CAN FD | CAN 2.0 | ≤8字节 | 1Mbps | ✅ 正常 |
| CAN FD | CAN 2.0 | >8字节 | 任意 | ❌ 错误帧 |
| CAN FD | CAN 2.0 | ≤8字节 | >1Mbps | ❌ 错误帧 |

## 6. C语言数据结构定义

```c
/* CAN FD 帧结构体 */
typedef struct {
    /* 标识符和帧类型 */
    uint32_t id;                /* 11位标准ID 或 29位扩展ID */
    uint8_t  is_extended;       /* 0=标准帧, 1=扩展帧 */
    uint8_t  is_fd;             /* 0=CAN 2.0, 1=CAN FD */

    /* CAN FD 特有 */
    uint8_t  brs;               /* Bit Rate Switch: 0=关闭, 1=开启 */
    uint8_t  esi;               /* Error State Indicator */

    /* 数据场 */
    uint8_t  dlc;               /* 数据长度码 (0-15) */
    uint8_t  data[64];          /* 数据 (CAN FD最大64字节) */

    /* 元数据 */
    uint32_t timestamp;         /* 接收时间戳 */
    uint8_t  filter_match;      /* 匹配滤波器索引 */
} CAN_FD_Frame;

/* CAN FD 控制器配置 */
typedef struct {
    /* 仲裁段波特率配置 */
    uint32_t nominal_bitrate;   /* 典型值: 500kbps, 1Mbps */
    uint16_t nominal_prescaler;
    uint8_t  nominal_tseg1;     /* 时间段1 */
    uint8_t  nominal_tseg2;     /* 时间段2 */
    uint8_t  nominal_sjw;       /* 同步跳转宽度 */

    /* 数据段波特率配置 */
    uint32_t data_bitrate;      /* 典型值: 2Mbps, 4Mbps, 8Mbps */
    uint16_t data_prescaler;
    uint8_t  data_tseg1;
    uint8_t  data_tseg2;
    uint8_t  data_sjw;

    /* 发送延迟补偿 */
    uint8_t  tdc_enable;        /* Transceiver Delay Compensation */
    uint8_t  tdc_offset;
} CAN_FD_Config;

/* 发送延迟补偿 (TDC) 说明:
 *
 * 在高波特率下，收发器的环路延迟会影响采样点。
 * TDC测量从发送到接收的延迟，自动调整采样点。
 *
 * 计算公式:
 * TDC_offset = 收发器延迟 + 收发器内部延迟
 * 典型值: 2-4个时间份额
 */
```

## 7. 性能优化分析

### 7.1 带宽效率计算

```
理论带宽效率分析:

场景: 传输64字节数据，比较CAN 2.0和CAN FD

CAN 2.0 (500kbps, 8字节×8帧):
├── 每帧开销: SOF(1) + ID(11) + Control(6) + CRC(15) + ACK(2) + EOF(7) = 42位
├── 每帧数据: 8字节 = 64位
├── 每帧填充位: 约10个 (估算)
├── 每帧总长度: 42 + 64 + 10 = 116位
├── 8帧总长度: 8 × 116 = 928位
└── 传输时间: 928 / 500k = 1856μs

CAN FD (500k仲裁, 2M数据, 单帧64字节):
├── 仲裁段: SOF(1) + ID(11) + Control(8) + 填充(~5) = 25位 @ 500kbps = 50μs
├── 数据段: DLC(4) + Data(512) + SBC(4) + 填充(~30) = 550位 @ 2Mbps = 275μs
├── CRC段: CRC21(21) + 填充(~5) = 26位 @ 500kbps = 52μs
├── ACK+EOF: 9位 @ 500kbps = 18μs
└── 总时间: 50 + 275 + 52 + 18 = 395μs

加速比: 1856μs / 395μs ≈ 4.7倍
```
