## 概述

IO操作是系统性能的关键瓶颈之一。不同的IO模型适用于不同的应用场景，理解它们的差异有助于选择最适合的方案。

---

## 1. 核心特性对比

| 方法 | 适用场景 | 复杂度 | 性能 | 资源占用 |
|------|----------|--------|------|----------|
| **阻塞IO** | 简单应用，低并发 | ⭐ 低 | 低（线程阻塞） | 高（一连接一线程） |
| **非阻塞IO** | 需要快速响应的场景 | ⭐⭐⭐ 高 | 中（轮询开销） | 中 |
| **IO多路复用** | 高并发网络服务 | ⭐⭐⭐ 高 | 高（事件驱动） | 低（单线程多连接） |
| **异步IO** | 高性能IO密集型应用 | ⭐⭐⭐⭐ 很高 | 很高（零拷贝） | 很低 |
| **内存映射IO** | 大文件处理，共享内存 | ⭐⭐ 中 | 很高（减少拷贝） | 中（内存占用） |

---

## 2. 实现机制对比

| IO模型 | 系统调用 | 通知机制 | 数据拷贝次数 | 典型应用 |
|--------|----------|----------|--------------|----------|
| 阻塞IO | read/write | 返回即完成 | 2（内核→用户） | 传统Web服务器 |
| 非阻塞IO | read/write + O_NONBLOCK | 立即返回EAGAIN | 2 | 简单轮询应用 |
| IO多路复用 | select/poll/epoll | 事件通知就绪 | 2 | Nginx, Redis |
| 异步IO | aio / io_uring | 完成回调/信号 | 1-2 | 数据库, 高频交易 |
| 内存映射 | mmap/munmap | 缺页中断 | 0-1 | 数据库索引, 图像处理 |

## 5. 代码示例对比

### 阻塞IO示例

```c
// 简单直接，但线程会阻塞
ssize_t n = read(fd, buf, sizeof(buf));
if (n > 0) {
    process_data(buf, n);
}
```

### IO多路复用示例（epoll）

```c
// 事件驱动，单线程处理多连接
int epfd = epoll_create1(0);
struct epoll_event ev = {.events = EPOLLIN, .data.fd = sockfd};
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev);
// 等待事件，而非等待数据
int nfds = epoll_wait(epfd, events, MAX_EVENTS, -1);
```

### 异步IO示例（io_uring）

```c
// 提交请求后无需等待，通过环形缓冲区接收完成通知
struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, size, offset);
io_uring_submit(&ring);
// 继续执行其他任务...
io_uring_wait_cqe(&ring, &cqe);  // 获取完成事件
```

## 选择建议

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 简单脚本/原型 | 阻塞IO | 开发效率优先 |
| <100并发连接 | 阻塞IO + 线程池 | 代码简单，足够应付 |
| 10K+并发连接 | epoll/kqueue | 成熟的C10K解决方案 |
| 超高性能存储 | io_uring | 最新Linux最优解 |
| 大文件随机访问 | 内存映射 | 减少数据拷贝，利用OS缓存 |
