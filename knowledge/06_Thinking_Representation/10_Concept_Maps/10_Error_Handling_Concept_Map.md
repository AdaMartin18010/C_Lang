## 3. 错误处理机制详解

### 3.1 返回值检查

```
[返回值错误处理]
    │
    ├──[整数返回]
    │       │
    │       ├──[0]──► [成功]
    │       ├──[>0]──► [警告/具体错误码]
    │       └──[<0]──► [错误]
    │
    ├──[指针返回]
    │       │
    │       ├──[非NULL]──► [成功，返回有效指针]
    │       └──[NULL]──► [失败]
    │
    ├──[布尔返回]
    │       │
    │       ├──[true/非0]──► [成功]
    │       └──[false/0]──► [失败]
    │
    └──[专用错误码]
            │
            ├──[枚举定义]──► [E_SUCCESS, E_INVALID, E_NOMEM...]
            │
            └──[结构化错误]──► [错误码+错误信息]

[示例]
    │
    ├──[fopen]──► [返回FILE*]──► [NULL表示失败]
    │
    ├──[malloc]──► [返回void*]──► [NULL表示失败]
    │
    ├──[main]──► [返回int]──► [0成功，非0错误码]
    │
    └──[strcmp]──► [返回int]──► [负数/0/正数表示比较结果]
```

### 3.2 errno机制

```
[<errno.h>错误处理]
    │
    ├──[errno]──► [线程局部变量]──► [存储最近错误码]
    │
    ├──[错误码定义]
    │       │
    │       ├──[EDOM]──► [定义域错误(数学)]
    │       ├──[ERANGE]──► [结果超出范围]
    │       ├──[EILSEQ]──► [非法序列(宽字符)]
    │       ├──[ENOMEM]──► [内存不足(POSIX)]
    │       ├──[EINVAL]──► [无效参数(POSIX)]
    │       └──[EIO]──► [IO错误(POSIX)]
    │
    ├──[相关函数]
    │       │
    │       ├──[strerror(errno)]──► [错误码转字符串]
    │       └──[perror("prefix")]──► [打印错误信息]
    │
    └──[使用模式]
            │
            ├──[调用可能失败的函数]
            │       │
            │       ├──[检查返回值]──► [表示失败]
            │       │
            │       └──[检查errno]──► [获取具体错误原因]
            │
            └──[注意]──► [仅当函数文档说明时才检查errno]

[errno使用示例]
    FILE *fp = fopen("file.txt", "r");
    if (fp == NULL) {
        // 错误检测
        fprintf(stderr, "Error: %s\n", strerror(errno));
        // 或
        perror("fopen failed");
        return -1;
    }
```

### 3.3 错误传播策略

```
[错误传播方式]
    │
    ├──[直接返回]
    │       │
    │       ├──[函数]──► [检测到错误]──► [立即返回错误码]
    │       │
    │       └──[优点]──► [简单直接]
    │
    ├──[错误链]
    │       │
    │       ├──[底层]──► [错误码A]
    │       ├──[中层]──► [包装为错误码B，保留A]
    │       └──[上层]──► [包装为错误码C，保留B、A]
    │
    ├──[异常模拟]
    │       │
    │       ├──[setjmp/longjmp]──► [非局部跳转]
    │       │
    │       └──[实现]──► [类似try-catch的跳转]
    │
    └──[回调通知]
            │
            ├──[注册错误处理器]──► [错误发生时调用]
            │
            └──[应用]──► [信号处理]──► [自定义错误响应]

[错误链示例]
    low_level() 返回 E_READ_ERROR
           │
           ▼
    mid_level() 检测到 E_READ_ERROR
           │
           └──包装为 E_FILE_ACCESS
           │
           ▼
    high_level() 检测到 E_FILE_ACCESS
           │
           └──包装为 E_OPERATION_FAILED
           │
           ▼
    用户看到的: "操作失败: 文件访问错误: 读取错误"
```

## 4. 详细命题表

### 4.1 返回值机制

| 编号 | 概念A | 连接词 | 概念B | 详细说明 |
|------|-------|--------|-------|----------|
| RV1 | 返回值 | 是 | 最常用错误传递方式 | C语言首选机制 |
| RV2 | 返回码 | 需要 | 文档说明 | 明确什么值表示错误 |
| RV3 | 返回NULL | 通常表示 | 指针函数失败 | 如malloc, fopen |
| RV4 | 返回负数 | 通常表示 | 整数函数失败 | 如系统调用 |
| RV5 | 返回值检查 | 是 | 调用者责任 | 忽略返回值是常见错误源 |
| RV6 | 多错误码 | 通过 | 不同返回值区分 | 如EAGAIN vs EINTR |

### 4.2 errno机制

| 编号 | 概念A | 连接词 | 概念B | 详细说明 |
|------|-------|--------|-------|----------|
| ER1 | errno | 是 | 线程局部变量 | C11起规定线程安全 |
| ER2 | errno | 由 | 系统调用和库函数设置 | 非所有函数都使用 |
| ER3 | errno | 在 | 函数成功时 | 可能被清除也可能保留 |
| ER4 | strerror | 将 | errno转为可读字符串 | 便于错误报告 |
| ER5 | perror | 打印 | errno对应的错误信息 | 到stderr |
| ER6 | 检查errno | 必须 | 在确认失败后 | 否则可能读到旧值 |

### 4.3 错误传播

| 编号 | 概念A | 连接词 | 概念B | 详细说明 |
|------|-------|--------|-------|----------|
| EP1 | 错误传播 | 可以通过 | 返回码层层传递 | 每层检查并返回 |
| EP2 | 错误传播 | 可以通过 | goto集中处理 | 清理资源后返回 |
| EP3 | 错误传播 | 可以通过 | setjmp/longjmp | 非局部跳转 |
| EP4 | 错误上下文 | 应该 | 逐层丰富 | 添加更多错误信息 |
| EP5 | 资源泄漏 | 在 | 错误处理时 | 需要特别注意清理 |
| EP6 | 部分失败 | 需要 | 回滚已完成的操作 | 保持系统一致性 |

### 4.4 恢复策略

| 编号 | 概念A | 连接词 | 概念B | 详细说明 |
|------|-------|--------|-------|----------|
| RS1 | 重试 | 适用于 | 临时性错误 | 如EAGAIN, EINTR |
| RS2 | 回滚 | 适用于 | 事务性操作 | 撤销已完成的步骤 |
| RS3 | 降级 | 适用于 | 可选功能失败 | 使用备用方案 |
| RS4 | 终止 | 适用于 | 不可恢复错误 | 如内存耗尽 |
| RS5 | 用户提示 | 适用于 | 输入错误 | 允许用户纠正 |
| RS6 | 日志记录 | 应该 | 记录所有错误 | 便于事后分析 |

## 5. 错误处理模式

### 5.1 错误处理编码模式

```
[模式1: 直接返回]
    int func() {
        if (error_condition) {
            return ERROR_CODE;
        }
        return SUCCESS;
    }

[模式2: goto清理]
    int func() {
        int result = 0;
        char *buf = malloc(100);
        if (!buf) {
            result = -ENOMEM;
            goto cleanup;
        }

        FILE *fp = fopen("file", "r");
        if (!fp) {
            result = -EIO;
            goto cleanup;
        }

        // 正常处理

    cleanup:
        if (fp) fclose(fp);
        if (buf) free(buf);
        return result;
    }

[模式3: 嵌套if]
    int func() {
        int result = 0;

        if (step1()) {
            if (step2()) {
                if (step3()) {
                    // 成功
                } else {
                    result = ERROR_STEP3;
                    undo_step2();
                }
                undo_step1();
            } else {
                result = ERROR_STEP2;
            }
            undo_step1();
        } else {
            result = ERROR_STEP1;
        }

        return result;
    }

[模式4: 宏封装]
    #define CHECK(call) \
        do { \
            int _err = (call); \
            if (_err < 0) { \
                LOG_ERROR("%s failed: %d", #call, _err); \
                return _err; \
            } \
        } while(0)

    int func() {
        CHECK(step1());
        CHECK(step2());
        CHECK(step3());
        return 0;
    }
```

### 5.2 结构化错误处理

```
[错误信息结构]
    │
    ├──[错误码]──► [机器可识别的错误标识]
    │
    ├──[错误信息]──► [人类可读的描述]
    │
    ├──[错误位置]──► [文件名、行号]
    │
    └──[错误链]──► [导致此错误的根本原因]

[实现示例]
    typedef struct Error {
        int code;
        const char *message;
        const char *file;
        int line;
        struct Error *cause;
    } Error;

    #define ERROR(code, msg) \
        (&(Error){(code), (msg), __FILE__, __LINE__, NULL})

    #define ERROR_CAUSE(code, msg, cause) \
        (&(Error){(code), (msg), __FILE__, __LINE__, (cause)})
```

## 8. 常见误区澄清

| 误区 | 正确理解 |
|------|---------|
| 可以忽略返回值 | 大多数C函数通过返回值报告错误，必须检查 |
| errno总是有效 | 只有函数文档说明时才检查errno |
| setjmp/longjmp是异常 | 它是不受控制的跳转，不调用析构函数 |
| assert用于运行时错误 | assert用于调试，不应处理可恢复错误 |
| 错误处理影响性能 | 正常路径不应有明显开销 |
| 所有错误都要恢复 | 编程错误应快速失败，而非尝试恢复 |
| 错误信息越详细越好 | 平衡安全性，避免泄露敏感信息 |
| 错误码越多越好 | 错误码应覆盖场景，但避免过度设计 |
| 错误处理代码不需要测试 | 错误处理路径同样需要测试 |
| 全局错误处理器足够 | 不同模块可能需要不同的处理策略 |

### 8.1 关键区分

```
[可恢复错误] vs [不可恢复错误]

可恢复错误:
- 外部因素导致（如文件不存在、网络中断）
- 程序可以采取措施恢复
- 示例: 打开文件失败、内存临时不足
- 处理: 重试、使用默认值、通知用户

不可恢复错误:
- 程序逻辑错误（如空指针解引用、数组越界）
- 继续执行可能导致未定义行为
- 示例: 断言失败、内存损坏
- 处理: 立即终止、生成core dump
```

```
[同步错误] vs [异步错误]

同步错误:
- 与特定操作直接相关
- 立即通过返回值报告
- 示例: fopen失败返回NULL

异步错误:
- 与操作无直接时序关系
- 通过信号或回调报告
- 示例: SIGSEGV、SIGALRM
```
