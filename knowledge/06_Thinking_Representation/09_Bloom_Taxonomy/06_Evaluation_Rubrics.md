# 评价标准与检查清单 (Evaluate Level)

> 本文档提供代码审查、设计评估、安全合规评估的完整评价标准和检查清单。

## 1. 代码审查检查清单

### 1.1 通用代码审查清单

#### 1.1.1 可读性与可维护性

**命名规范**

- [ ] 变量名清晰表达用途 (studentCount而非n)
- [ ] 函数名使用动词+名词 (getStudentName)
- [ ] 常量全大写下划线分隔 (MAX_BUFFER_SIZE)
- [ ] 类型名使用大驼峰命名 (StudentInfo)
- [ ] 布尔变量使用肯定语气 (is_valid)

**代码组织**

- [ ] 函数长度控制在50行以内
- [ ] 单行代码不超过80/120字符
- [ ] 缩进一致 (4空格)
- [ ] 空行用于逻辑分组

**注释质量**

- [ ] 函数有文档注释(功能/参数/返回值)
- [ ] 复杂算法有解释性注释
- [ ] 边界情况有说明
- [ ] 避免无意义注释

#### 1.1.2 正确性与健壮性

**边界条件**

- [ ] 处理空输入(NULL指针、空字符串)
- [ ] 处理最大/最小值边界
- [ ] 数组访问检查索引范围
- [ ] 除法检查除数为0
- [ ] 递归有终止条件

**错误处理**

- [ ] 检查系统调用返回值
- [ ] 错误处理路径完善
- [ ] 错误信息清晰有用

**数据验证**

- [ ] 外部输入经过验证
- [ ] 字符串长度有限制
- [ ] 指针检查NULL

#### 1.1.3 性能与效率

**算法效率**

- [ ] 算法时间复杂度最优
- [ ] 无重复计算
- [ ] 递归可改为迭代

**资源使用**

- [ ] 内存分配必要
- [ ] 无内存泄漏风险
- [ ] 文件句柄及时关闭

**缓存友好**

- [ ] 考虑缓存局部性
- [ ] 循环访问模式连续

### 1.2 并发代码审查

- [ ] 共享数据有同步保护
- [ ] 锁粒度合适
- [ ] 锁获取顺序全局一致
- [ ] 复合操作原子保护
- [ ] 条件变量用while检查

### 1.3 内存管理审查

- [ ] malloc/free配对使用
- [ ] 异常路径释放内存
- [ ] 指针所有权清晰
- [ ] 不返回局部变量地址
- [ ] 释放后指针置NULL

## 3. 安全合规评估

### 3.1 MISRA C:2023 关键规则

**语言特性限制**

- [ ] 避免动态内存分配 (Dir 4.12)
- [ ] 避免递归 (Dir 4.13)
- [ ] 避免setjmp/longjmp (Rule 17.13)
- [ ] 避免浮点数比较 (Rule 14.4)

**类型系统**

- [ ] 显式类型转换 (Rule 10.8)
- [ ] 避免隐式类型窄化 (Rule 10.3)
- [ ] 枚举完整处理 (Rule 16.4)

**指针使用**

- [ ] 避免void指针算术 (Rule 18.5)
- [ ] 检查NULL指针 (Rule 17.5)
- [ ] 指针偏移检查边界 (Rule 18.1)

**控制流**

- [ ] 循环有确定上界 (Rule 14.4)
- [ ] switch有default (Rule 16.4)
- [ ] 避免goto (Rule 15.1)

### 3.2 CERT C安全编码

**字符串操作**

- [ ] 不使用strcpy/strcat，改用strncpy_s
- [ ] sprintf检查缓冲区，改用snprintf
- [ ] 不使用gets，使用fgets
- [ ] 字符串以null结尾

**内存管理**

- [ ] 双重释放检查 (MEM30-C)
- [ ] 释放后使用检查 (MEM30-C)
- [ ] 敏感数据清零 (MSC06-C)

**整数操作**

- [ ] 整数溢出检查 (INT30-C)
- [ ] 除零检查 (INT33-C)
- [ ] 移位操作检查 (INT34-C)

### 3.3 安全漏洞评估矩阵

| 漏洞类型 | CWE编号 | 防护策略 |
|----------|---------|----------|
| 缓冲区溢出 | CWE-121 | 边界检查、安全函数 |
| 整数溢出 | CWE-190 | 范围检查、宽类型 |
| 格式化字符串 | CWE-134 | 硬编码格式字符串 |

| 双重释放 | CWE-415 | 统一释放逻辑 |

## 4. 质量评分标准

### 4.1 代码质量评级

| 等级 | 检查项通过率 | 要求 |
|------|-------------|------|
| A (优秀) | >=95% | 无严重问题，架构良好 |
| B (良好) | 80-94% | 少量轻微问题 |
| C (合格) | 60-79% | 无严重问题，有改进空间 |
| D (需改进) | 40-59% | 存在多个问题 |
| F (不合格) | <40% | 严重问题多，需重写 |

### 4.2 评分细则

**功能性 (25分)**

- 需求实现完整 (10分)
- 边界条件处理 (8分)
- 错误处理完善 (7分)

**可靠性 (20分)**

- 异常处理 (7分)
- 资源管理 (7分)
- 并发安全 (6分)

**效率 (15分)**

- 时间复杂度 (8分)
- 空间复杂度 (7分)

**可维护性 (20分)**

- 代码可读性 (7分)
- 文档完整性 (7分)
- 测试覆盖 (6分)

**可移植性 (10分)**

- 标准合规 (5分)
- 平台无关 (5分)

**安全性 (10分)**

- 输入验证 (5分)
- 安全函数使用 (5分)

## 5. 评估流程模板

### 5.1 代码审查会议流程

```
1. 准备阶段
   - 作者提交代码和文档
   - 审查者独立审查(提前)
   - 准备问题清单

2. 审查会议
   - 作者概述变更(5分钟)
   - 审查者提问(20分钟)
   - 讨论问题严重程度(15分钟)
   - 确认修改项(10分钟)

3. 后续跟踪
   - 作者修改代码
   - 审查者验证修改
   - 确认关闭问题
```

### 5.2 问题分级标准

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|----------|------|
| P0 (阻塞) | 必须修复才能合并 | 立即 | 安全漏洞、崩溃 |
| P1 (严重) | 强烈建议修复 | 24小时 | 内存泄漏、竞态条件 |
| P2 (一般) | 建议修复 | 本周 | 代码风格、轻微优化 |
| P3 (轻微) | 可选修复 | 下次迭代 | 文档改进 |
