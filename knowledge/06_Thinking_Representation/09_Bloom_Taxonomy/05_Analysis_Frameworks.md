# 分析框架指南 (Analyze Level)

> 本文档提供代码分析、性能分析、调试分析的系统化框架和决策树，帮助建立深度分析能力。

---

## 1. 代码分析方法框架

### 1.1 静态代码分析框架

#### 1.1.1 代码审查检查清单

```
【架构层面】
□ 模块职责是否单一明确？
□ 接口设计是否遵循最小知识原则？
□ 依赖关系是否形成有向无环图？
□ 是否使用了合适的设计模式？
□ 错误处理策略是否一致？

【代码层面】
□ 变量命名是否清晰表达意图？
□ 函数长度是否控制在50行以内？
□ 圈复杂度是否超过10？
□ 是否存在重复代码？
□ 魔法数字是否有具名常量？

【安全层面】
□ 所有输入是否经过验证？
□ 缓冲区操作是否检查边界？
□ 指针使用前是否检查NULL？
□ 资源释放是否与获取匹配？
□ 是否存在整数溢出风险？

【性能层面】
□ 算法复杂度是否最优？
□ 是否存在不必要的内存拷贝？
□ 热点路径是否有明显低效代码？
□ 缓存局部性是否被考虑？
```

#### 1.1.2 数据流分析框架

```
步骤1: 识别输入源
├── 函数参数
├── 全局变量
├── 文件/网络输入
└── 环境变量

步骤2: 追踪数据流向
├── 赋值操作
├── 函数传递
├── 类型转换
└── 格式化操作

步骤3: 识别敏感操作点
├── 内存访问
├── 系统调用
├── 外部库调用
└── 权限操作

步骤4: 验证约束条件
├── 范围检查
├── 类型检查
├── 空值检查
└── 权限检查
```

#### 1.1.3 控制流分析框架

```
工具: CFG (Control Flow Graph)

分析步骤:
1. 绘制基本块划分
   ┌─────────┐
   │  Entry  │
   └────┬────┘
        ▼
   ┌─────────┐
   │  B1     │───条件分支───┐
   └────┬────┘              │
        │True               │False
        ▼                   ▼
   ┌─────────┐         ┌─────────┐
   │  B2     │         │  B3     │
   └────┬────┘         └────┬────┘
        │                    │
        └────────┬───────────┘
                 ▼
            ┌─────────┐
            │  B4     │
            └────┬────┘
                 ▼
            ┌─────────┐
            │  Exit   │
            └─────────┘

2. 识别关键路径
   - 最长路径(时间复杂度高)
   - 最短路径(快速返回)
   - 错误处理路径

3. 分析循环结构
   - 循环边界条件
   - 循环不变量
   - 循环展开机会

4. 识别不可达代码
```

### 1.2 动态代码分析框架

#### 1.2.1 执行路径追踪

```
方法1: 代码插桩
#define TRACE() printf("%s:%d\n", __FILE__, __LINE__)
#define ENTER_FUNC() printf("Enter %s\n", __func__)
#define EXIT_FUNC() printf("Exit %s\n", __func__)

方法2: 使用调试器
- GDB: set logging on, break + commands
- LLDB: thread trace

方法3: 使用工具
- strace: 系统调用追踪
- ltrace: 库函数追踪
- ftrace: 内核函数追踪(Linux)
```

#### 1.2.2 内存访问分析

```
工具链:
┌─────────────────────────────────────────┐
│  Valgrind (memcheck, cachegrind)       │
│  AddressSanitizer (ASan)               │
│  MemorySanitizer (MSan)                │
│  Intel VTune                           │
│  perf mem                              │
└─────────────────────────────────────────┘

分析维度:
1. 内存泄漏检测
   - 分配未释放
   - 丢失指针引用
   - 循环引用(如适用)

2. 越界访问检测
   - 栈越界
   - 堆越界
   - 全局变量越界

3. 非法内存访问
   - 使用已释放内存(use-after-free)
   - 返回局部变量地址
   - 双重释放

4. 未初始化使用
   - 变量未初始化
   - 内存未清零
```

### 2.2 CPU性能分析框架

#### 2.2.1 热点分析流程

```
步骤1: 收集性能数据
   $ perf record -g ./program
   $ perf report

步骤2: 识别热点函数
   - 按CPU时间排序
   - 关注占比>5%的函数
   - 查看调用栈

步骤3: 分析热点原因
   ┌────────────────────────────────────────┐
   │  可能性分析:                           │
   │  □ 算法复杂度过高？                    │
   │  □ 循环效率低？                        │
   │  □ 缓存未命中率高？                    │
   │  □ 分支预测失败？                      │
   │  □ 频繁内存分配？                      │
   │  □ 锁竞争激烈？                        │
   └────────────────────────────────────────┘

步骤4: 优化并验证
   - 实施优化
   - 重新测量
   - 计算加速比
```

#### 2.2.2 缓存性能分析

```
工具: perf stat -e cache-misses,cache-references,L1-dcache-load-misses

分析指标:
┌──────────────────┬──────────────┬────────────────┐
│ 指标             │ 健康范围     │ 优化方向       │
├──────────────────┼──────────────┼────────────────┤
│ L1命中率         │ >95%         │ 数据局部性     │
│ LLC(末级缓存)    │ >90%         │ 工作集大小     │
│ 缓存未命中/指令  │ <0.05        │ 预取、对齐     │
│ TLB未命中        │ <1%          │ 大页、分页优化 │
└──────────────────┴──────────────┴────────────────┘

优化策略:
1. 数据局部性
   - 结构体字段重排
   - 数组合并(AoS vs SoA)
   - 循环分块(tiling)

2. 预取优化
   - 软件预取指令
   - 硬件预取友好访问模式

3. 伪共享避免
   - 填充到缓存行边界
   - 线程私有数据
```

### 2.3 内存性能分析框架

#### 2.3.1 内存使用分析

```
分析维度:

1. 内存分配模式
   $ valgrind --tool=massif ./program
   $ ms_print massif.out.*

   关注:
   - 峰值内存使用
   - 分配频率
   - 分配大小分布

2. 分配热点识别
   - 哪些代码路径分配最多？
   - 是否存在分配抖动？
   - 小对象分配是否集中？

3. 优化策略决策树
              【内存优化】
                    │
      ┌─────────────┼─────────────┐
      ▼             ▼             ▼
  [分配频繁]    [内存碎片]    [内存泄漏]
      │             │             │
      ▼             ▼             ▼
  -内存池       -伙伴系统      -智能指针
  -对象池       -slab分配器    -RAII模式
  -栈分配       -压缩整理      -静态分析
```

#### 2.3.2 内存泄漏追踪流程

```
步骤1: 检测泄漏
   $ valgrind --leak-check=full --show-leak-kinds=all ./program

步骤2: 分类泄漏
   - definitely lost: 确定泄漏
   - indirectly lost: 间接泄漏
   - possibly lost: 可能泄漏(指针偏移)
   - still reachable: 程序结束仍可访问(通常可接受)

步骤3: 定位泄漏点
   - 查看分配调用栈
   - 分析生命周期
   - 检查释放路径

步骤4: 修复验证
   - 添加释放代码
   - 重新运行检测
   - 确认泄漏消失
```

### 2.4 IO性能分析框架

```
磁盘IO分析:
$ iostat -xz 1
$ pidstat -d 1
$ blktrace -d /dev/sda

关注指标:
- await: 平均IO等待时间(>10ms需关注)
- %util: 设备利用率(接近100%表示饱和)
- svctm: 服务时间(>5ms表示磁盘慢)
- 队列深度: >2表示可能有排队

网络IO分析:
$ ss -tan
$ nicstat 1
$ tcpdump/wireshark

关注指标:
- 重传率: >1%表示网络问题
- 接收缓冲区溢出
- 连接状态分布
- RTT变化
```

### 3.2 崩溃分析框架

#### 3.2.1 段错误(SIGSEGV)分析

```
常见原因:
1. 空指针解引用
   - 调试: gdb中p ptr，检查是否为0x0
   - 修复: 添加NULL检查

2. 野指针
   - 调试: 检查指针值是否有效地址
   - 工具: AddressSanitizer
   - 修复: 初始化指针，释放后置NULL

3. 栈溢出
   - 调试: 检查大数组分配，递归深度
   - 工具: -fstack-protector-strong
   - 修复: 大数组用malloc，递归改循环

4. 堆损坏
   - 调试: valgrind, ASan
   - 检查: 越界写入、重复释放
   - 修复: 使用安全函数

5. 使用已释放内存
   - 调试: ASan检测use-after-free
   - 修复: 释放后置NULL，使用智能指针

分析步骤:
1. 获取崩溃地址
2. 检查地址特征(是否NULL、栈区、堆区)
3. 查看寄存器和内存状态
4. 回溯最近的操作
```

#### 3.2.2 死锁分析

```
检测工具:
- GDB: thread apply all bt
- helgrind (valgrind)
- 锁分析器(编译器支持)

分析步骤:
1. 确认死锁
   - 程序无响应
   - 线程都在等待锁

2. 获取线程状态
   (gdb) info threads
   (gdb) thread apply all bt

3. 分析等待图
   示例死锁模式:
   Thread 1: 持有Lock A，等待Lock B
   Thread 2: 持有Lock B，等待Lock A

4. 解决方案
   - 锁顺序一致性
   - 超时获取(trylock)
   - 锁层级检测
   - 无锁数据结构
```

### 3.4 并发问题调试框架

#### 3.4.1 数据竞争检测

```
工具:
- ThreadSanitizer (TSan): -fsanitize=thread
- helgrind
- Intel Inspector

分析输出:
WARNING: ThreadSanitizer: data race
  Write of size 4 at 0x7fff by thread T2:
    #0 func2 race.c:20
  Previous write of size 4 at 0x7fff by thread T1:
    #0 func1 race.c:10

修复方案:
1. 使用互斥锁保护
2. 使用原子操作
3. 使用线程局部存储
4. 重新设计避免共享
```

#### 3.4.2 原子性问题分析

```
检查清单:
□ 复合操作是否原子？
  - i++ 不是原子操作
  - 64位赋值在32位系统可能非原子

□ 内存序是否足够？
  - 多线程flag需要atomic
  - 可能需要memory_order_acquire/release

□ 检查-然后-行动是否安全？
  - 非原子检查+行动 = 竞态条件
  - 应使用compare-and-swap

工具:
- 代码审查
- TSan检测
- 压力测试
```

### 4.2 根因分析模板 (5 Whys)

```
问题: 服务在高负载下崩溃

为什么崩溃？→ 段错误在日志写入模块
为什么段错误？→ 缓冲区指针为NULL
为什么指针为NULL？→ 内存分配失败返回NULL未检查
为什么分配失败？→ 内存泄漏导致OOM
为什么有内存泄漏？→ 异常路径未释放资源

根因: 异常处理不完善导致资源泄漏
解决方案: 使用RAII模式，添加异常安全代码
```
